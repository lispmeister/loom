# Architecture Review by GPT-5.2-Codex

## Summary
The project implements the three-part architecture described in README: a Supervisor on the host to manage containers and lineage, a Prime agent to run the tool loop and self-modify, and ephemeral Lab containers to execute `program.md` and commit changes. The code is coherent and aligned with the docs: tools are wired, self-modify actions exist, and the autonomous loop (reflect → spawn → verify → promote/rollback) is implemented with stop conditions and a fitness log. The system is already operational and minimal, but there are brittle edges in verification, state recovery, and enforcement of Lab constraints.

## Strengths
- Clear separation of responsibilities across Supervisor/Prime/Lab and clean entry points.
- Self-modify tools are integrated into the Prime tool registry and are absent from Labs as intended.
- Verification includes both tests and an LLM diff review, with results stored in generation reports.
- Autonomous loop has explicit stopping conditions, a fitness log, and a reflect step based on recent history and priorities.
- Codebase is small and readable; the tool-use loop is explicit and easy to reason about.

## Architectural Risks and Gaps
1. **Supervisor state recovery**
   - Active lab state (timeouts/pollers) lives only in memory. A Supervisor restart loses in-flight state and does not reconcile `generations.edn` with real container state.
   - No boot-time reconciliation or cleanup path exists.

2. **Verification coupling and fragility**
   - `verify_generation` returns a string and uses a shared atom (`last-verification`) for promote/rollback metadata. This can go stale if multiple verifications run or if verification fails mid-way.
   - Autonomous parsing relies on regexes over string output and will break with formatting changes.

3. **Lab constraints are advisory only**
   - Labs are instructed not to run tests/builds, but `bash` remains available. Nothing enforces the constraint.
   - If a Lab ignores the prompt, it can still run forbidden commands.

4. **Error handling and typed results**
   - Many APIs return strings on error instead of structured data. This makes automation and error routing brittle.
   - `read-json-body` assumes valid JSON and can throw on malformed requests.

5. **Container port allocation**
   - Host port is computed as `18400 + gen-num`, which can collide on long runs, concurrent Supervisors, or large generation numbers.

6. **Fitness function incentives**
   - The fitness score rewards test count and assertions, which can be gamed without improving system capability.

7. **Merge safety**
   - Promotions use plain `git merge` without conflict handling or `--ff-only`, risking conflicted worktrees on failed merges.

## Implementation Critique (Concrete)
- `verify_generation` uses git checkout and always returns to master, but on checkout failure it returns a string and does not update `last-verification`. Promote/rollback may then include stale verification metadata.
- Autonomous loop’s `parse-verify-result` is brittle and depends on output formatting; a structured return would be more reliable.
- The Prime loop’s context trimming inserts a marker string but does not produce a summary, which weakens long-running sessions.
- The Lab worker marks “no changes” as failure, which is correct for self-modification but makes doc-only tasks effectively impossible unless reflect avoids them.

## Suggested Priorities
- Add Supervisor boot-time reconciliation and safe cleanup for in-flight labs.
- Return structured verification results and remove the `last-verification` atom coupling.
- Enforce Lab constraints at the tool layer (e.g., restrict or wrap `bash`).
- Harden HTTP request parsing and return structured errors.
- Improve port allocation strategy or delegate to the container runtime.
- Strengthen fitness signals to align with actual behavioral improvements.
