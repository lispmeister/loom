## Original Goals

Loom set out to build a **recursive self-improving coding agent** — one that can rewrite its own source code, test the modifications in an isolated container, and promote successful changes. The key insight: use Apple's VM-level containerization as a keep/revert boundary so the agent can experiment safely.

## What the MVP Achieved

Gen-1 succeeded on 2026-03-15: the Lab added line numbers to `read_file`, Prime verified the tests pass, and the change was merged to main. That's a real self-modification cycle. The codebase has a healthy test-to-source ratio (~1:1 by LOC).

## Critical Findings

### 1. The proof is thin — one generation isn't a pattern

One git tag (`gen-1`), one successful cycle. An outsider would say "that could be a fluke." To be convincing: at least 3–5 successful generations with varying task complexity (pure edit, refactor, new tool, bug fix). The story right now is "it worked once." It needs to be "it works reliably."

**Impact:** High — this is the credibility gap.
**Action:** Run 3–5 more generations after cleanup.

### 2. Three Malli schemas are defined but never used

`Proposal`, `ProbeResult`, and `Verdict` in `schemas.cljs` are declared as "fixed-point contracts" but the system doesn't use them. The real contract is `POST /spawn` with a `program_md` string and `POST /promote` with a generation number — plain JSON, unvalidated.

The conceptual framework of "mutable code + immutable contracts" relies on those contracts being load-bearing. Right now they're decorative.

**Impact:** Medium — conceptual integrity issue.
**Action:** Either wire them into the actual HTTP handlers or remove them and document the real contracts.

### 3. The eval server and eval client are dead code

`shared/eval_client.cljs` (39 LOC) and `lab/eval_server.cljs` (107 LOC) are compiled but never called in the self-modification flow. The `:lab` build target produces `out/lab.js` but Labs run `out/lab-worker.js`. The eval client was meant for Prime to probe Lab state, but `verify_generation` uses `git checkout + npm test` instead.

146 LOC of dead code in a 2,200 LOC codebase (~7%). More importantly, the README described the Lab as having a "Form eval server" which misrepresents what Labs actually do.

**Impact:** Medium — code clutter + doc confusion.
**Action:** Remove eval client. Keep eval server source but remove `:lab` build target. Update docs.

### 4. `verify_generation` shells out to raw git — fragile

`self_modify.cljs:117–159` does `git checkout branch` → `npm test && node out/test.js` → `git checkout master` on the *host repo*. If anything goes wrong between checkout and return-to-master (crash, unhandled rejection), the host repo is left on the lab branch. No try/finally equivalent — the `.then` chain assumes each step completes.

**Impact:** High — can leave the host repo in a broken state.
**Action:** Wrap in a robust finally-always-checkout pattern.

### 5. Lab worker uses `git add -A` — no selectivity

`worker.cljs:29` runs `git add -A && git commit`. If the Lab creates temp files, debug output, or anything unexpected, it all gets committed. `.gitignore` mitigates `lab-worker.js` and `program.md`, but any *other* artifact gets swept in.

**Impact:** Low — mitigated by `.gitignore`, but still a cleanliness issue.
**Action:** Consider `git add` with explicit paths or extend `.gitignore`.

### 6. No observability of what the Lab actually changed

`verify_generation` only checks "did tests pass." It has no visibility into *what changed*. No `git diff` in the verification flow. Prime promotes blindly if tests are green.

**Impact:** High — an outsider would ask "how do you know the Lab didn't introduce something harmful that passes tests?"
**Action:** Add `git diff --stat` (and optionally full diff) to `verify_generation` output. Prime should see the diff before promoting.

### 7. Conversation history grows unboundedly

`loop.cljs` appends every message to `:messages` and replays the full history on every Claude API call. For a 25-iteration tool-use loop, that's 50+ messages replayed, each containing tool results (which can be large file contents). This is the root cause of the rate-limit issue from gen-1.

**Impact:** High — scalability blocker.
**Action:** Already tracked as bead `loom-ajq.2`. Prioritize for cleanup.

### 8. `supervisor/http.cljs` is 307 lines doing too many things

Handles dashboard HTML, SSE, spawn, promote, rollback, workspace cleanup, and program.md persistence. The spawn handler alone is 60 lines.

**Impact:** Low — legibility issue, not a bug.
**Action:** Extract `cleanup-lab-workspace` and `save-program-md` into `supervisor/lab.cljs`.

### 9. Promise chains are hard to follow

`verify_generation` is a 50-line `.then` chain. Each step is anonymous. Extracting named functions for each step would significantly improve readability.

**Impact:** Low — legibility.
**Action:** Refactor long `.then` chains into named step functions.

### 10. No generation metrics or reports

`program.md` is saved to `tmp/programs/gen-N.md`, but there's no record of: what changed (diff), how long it took, how many API calls, tokens consumed, whether it retried. This data is needed to evaluate and improve the system.

**Impact:** Medium — needed for data-driven improvement.
**Action:** Create a `gen-N-report.json` with lifecycle metadata.

## What's Working Well

- **Clean separation of concerns** — Agent, Supervisor, Lab are genuinely independent.
- **Strong testing** — 90 tests, 0 failures; async/promise handling is solid.
- **Minimal dependencies** — Only Malli + shadow-cljs. No frameworks.
- **Explicit error handling** — Errors resolve as `{:error true :message "..."}`, never thrown.
- **Git as source of truth** — Branches, tags, lineage tracked durably.
- **Container isolation** — Labs run in VMs with no shared state with Prime.
- **Compact codebase** — 2,214 LOC for a working self-modifying agent is impressively lean.
