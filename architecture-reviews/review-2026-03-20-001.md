## Executive Summary

Loom proved that an LLM-powered agent can autonomously modify its own source code, verify the changes, and promote them to production — all without human intervention. It also proved that **source-code-level evolution is the wrong abstraction** for recursive self-improvement. This review extracts the learnings and closes the book on Loom as a standalone project. The successor, CAMBRIAN, will apply these lessons at the specification level.

---

## What Worked

### 1. The Three-Component Architecture

Prime/Supervisor/Lab separation was the right call. Each component has a clear, testable contract:

- **Prime** decides *what* to do (reflect, propose, verify, promote/rollback)
- **Supervisor** manages *where* it happens (container lifecycle, git branches, port allocation)
- **Lab** does *the work* (reads program.md, executes autonomously, commits results)

This separation survived 72 generations without architectural changes. It should carry forward into CAMBRIAN.

### 2. Containers as the Keep/Revert Boundary

Apple Containerization gave us VM-level isolation with fast startup. The mental model — "promote = merge + tag, revert = destroy + discard" — was simple and reliable. Zero data loss incidents. Zero container escapes.

The 5/5 stability run (gen-13–17) proved the infrastructure was solid before we attempted any real self-modification.

### 3. Two-Stage Verification

Tests alone were insufficient. The LLM code review caught issues that tests missed (e.g., style regressions, unnecessary changes, container artifacts in diffs). Requiring both gates eliminated false promotions.

### 4. program.md as Task Contract

The structured task contract — description, context, acceptance criteria, constraints — was the right interface between "decide what to do" and "do it." It's human-readable, LLM-parseable, and versionable. The Lab doesn't need to understand the codebase history; it just reads the contract and executes.

### 5. Append-Only Audit Trail

`generations.edn`, `fitness-log.jsonl`, `lessons-log.jsonl` — every decision was recorded. This made debugging autonomous runs straightforward: you could always trace back *why* a generation failed or succeeded.

### 6. Multi-Provider LLM Support

Split-provider configuration (different API keys, base URLs, and models for Prime vs Lab) proved essential for cost management and capability testing. Haiku for cheap experiments, Opus for serious modifications.

## What Didn't Work

### 1. Source-Code-Level Evolution Accumulates Cruft

**This is the core finding.** Each generation produces a diff against the current codebase. Over time:

- Diffs interact unpredictably. Gen-5's refactor may conflict with Gen-8's feature.
- Code style drifts as different LLMs (Haiku, Sonnet, Opus, Minimax) write different idioms.
- Dead code accumulates — no generation has incentive to clean up previous generations' work.
- The codebase becomes increasingly *path-dependent*: the order of promotions matters.

In 72 generations, only 1 was promoted. That's a 1.4% success rate. The promoted change was small (making 2 private functions public + adding 18 test assertions). Larger changes — the kind needed for meaningful improvement — have higher failure rates because they interact with more existing code.

**Conclusion:** Evolving source code via patches is viable for small, additive changes. It cannot sustain recursive self-improvement over many generations.

### 2. Model Capability Is the Bottleneck

| Model | Generations | Promoted | Failure Mode |
|-------|-------------|----------|--------------|
| Haiku | 10+ | 0 | Gets close but produces buggy async/mock code |
| Minimax M2.5 | 3 | 0 | Reads files but never writes — model-level limitation |
| Opus 4.6 | 1 | 1 | First-try success, clean code |

The gap between "can write code" and "can write correct ClojureScript that passes two-stage verification" is enormous. Only the most capable (and expensive) model succeeded. This means:

- Evolution at the code level requires expensive models, making birth costs high
- Cheaper models can't generate viable offspring, so you can't amortize cost through volume
- The fitness landscape is nearly flat: most random modifications fail, few are improvements

### 3. Reflect Step Has Weak Signal

The reflect step reads generation history, fitness scores, and codebase source to propose the next improvement. In practice:

- The fitness function is dominated by test count (easy to game by adding trivial tests)
- Token efficiency penalizes thorough work (longer conversations = lower score)
- There's no way to measure "is this code actually better?" without task benchmarks
- The LLM's self-assessment ("what's weakest?") is unreliable without ground truth

The reflect step works mechanically but doesn't produce meaningfully directed improvements.

### 4. Fitness Function Doesn't Capture What Matters

The v0 fitness formula: `(tests-run × 10) + (assertions × 1) − (total-tokens / 1000)`

Problems:
- **Goodhart's Law:** Optimizing for test count incentivizes trivial tests
- **No task performance:** Doesn't measure whether the agent can actually accomplish useful work
- **Token penalty is backwards:** Sometimes more tokens = better work (thorough analysis)
- **No code quality metric:** Clean, maintainable code scores the same as spaghetti

A real fitness function needs external task benchmarks: "can this agent solve Problem X?"

### 5. Lab Build Constraints

Labs couldn't run `npm test` because shadow-cljs compilation took ~25s inside the container VM, eating into the 5-minute timeout. This forced all verification to happen host-side (Prime), which:

- Added complexity to the verify pipeline
- Meant Labs couldn't self-validate before reporting "done"
- Made the feedback loop slower (Lab reports done → Prime fetches branch → Prime runs tests)

### 6. Infrastructure Bugs Dominated Early Generations

6 infrastructure bugs were found and fixed in the first 17 generations:

1. Branch fetch race condition
2. Local changes blocking checkout
3. .gitignore duplication
4. program.md gitignored incorrectly
5. LLM reviewer too strict on environment artifacts
6. Timeout handler missing branch fetch

These are not evolution bugs — they're plumbing bugs. The first ~20 generations were spent debugging infrastructure, not improving the agent. This suggests that the code-level approach has high fixed costs before evolution can even begin.

## Why CAMBRIAN Is the Right Next Step

### The Core Insight

Loom evolves **code** (phenotype). CAMBRIAN will evolve **specifications** (genotype). This is the key difference:

| | Loom (code evolution) | CAMBRIAN (spec evolution) |
|---|---|---|
| **Unit of evolution** | Source file diffs | Specification document |
| **Accumulation** | Cruft, path dependence | Clean regeneration each generation |
| **Search space** | All possible code modifications | Structured spec mutations |
| **Failure mode** | Subtle bugs from interacting diffs | Complete non-viability (binary) |
| **Build cost** | Incremental (cheap per gen) | Full rebuild (expensive per gen, but cleaner) |
| **Verification** | Tests + LLM review of diff | Tests + full acceptance criteria |

The analogy: Loom is Lamarckian evolution (inherit acquired traits — merge code patches). CAMBRIAN is Darwinian evolution (inherit genetic information — mutate specs, regenerate from scratch).

### What Carries Forward

1. **Three-component architecture** — Prime/Supervisor/Lab roles are sound
2. **Container isolation** — Keep/revert boundary works
3. **program.md contract pattern** — Generalizes to "spec as contract"
4. **Two-stage verification** — Tests + LLM review
5. **Append-only audit trail** — Generation history, fitness, lessons
6. **Multi-provider LLM support** — Essential for model diversity
7. **Stopping conditions** — Generation cap, token budget, plateau detection, signal handling

### What Changes

1. **No more code diffs** — Labs verify artifacts, they don't author code
2. **Spec is the genome** — ~300 lines vs ~2,600 lines of code
3. **Full regeneration** — Every generation starts from the spec, not from the previous code
4. **Artifact manifests** — Hash-verified, immutable generation artifacts
5. **Viability reports** — Structured binary outcome (viable/non-viable), not fitness gradients
6. **Spec-level lineage** — Track spec mutations, not code changes

### What Must Be Solved

1. **Fitness function tied to task performance** — Not test count, not token efficiency. Can the agent *do useful work*?
2. **Spec mutation strategy** — LLM-guided, failure-directed, crossover, or combination
3. **Birth cost management** — Full regeneration is expensive; need cheap viability pre-checks
4. **Spec format design** — Must be modular enough for crossover, structured enough for LLM parsing

## Recommendations for CAMBRIAN Implementation

1. **Start fresh.** Don't try to bolt CAMBRIAN onto Loom's codebase. The contracts are different (artifact manifests vs code diffs), the data model is different (spec lineage vs generation history), and the verification logic is fundamentally different (viability checks vs test-on-diff).

2. **Keep CAMBRIAN-SPEC-001 as the seed.** It's already a working generative specification (~400 lines). Test whether an LLM can produce a working agent from it before building any infrastructure.

3. **Define task benchmarks before building evolution.** Without external fitness criteria, evolution will optimize for reproduction cost, not usefulness. Define 10-20 concrete tasks the agent should be able to perform.

4. **Budget $50-100 for Phase 1 (Spec Genesis).** This is the validation step: can an LLM reliably regenerate a working agent from a spec? If not, the rest of CAMBRIAN doesn't matter.

5. **Use Loom's test suite as the minimum viability bar.** A CAMBRIAN-generated organism must pass 236 tests / 607 assertions to be considered viable. This gives a concrete, measurable birth criterion.

## Final State of Loom v0.2.0

This release represents the endpoint of source-code-level evolution. The codebase is functional:

- 236 tests, 607 assertions (5 environment-specific failures in container networking tests — not code bugs)
- Full pipeline operational: reflect → spawn → work → verify → promote/rollback
- Autonomous loop with stopping conditions
- Multi-provider LLM support
- CLI tools for individual operations
- HTML dashboards on all components
- Comprehensive generation history and fitness tracking

The architecture is sound. The lesson is that the right level of abstraction for evolution is the specification, not the source code.
