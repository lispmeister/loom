# Architecture Review — 2026-03-18-001

**Reviewer:** Claude Opus 4.6
**Scope:** CAMBRIAN proposal — evolutionary spec-based reproduction as successor architecture to Loom's code-merge model
**Status:** Proposal review (no code changes)

---

## Proposal Summary

CAMBRIAN proposes replacing Loom's current self-modification cycle (spawn Lab, modify code, promote back to Prime) with an evolutionary model where:

1. A **one-shot specification** is the genetic code of each agent
2. Primes produce successors **from scratch** using a mutated spec + LLM (no code merging)
3. Primes have **lifetime limits** and must promote a successor before death
4. **Resource constraints** (token budgets, compute, storage) act as natural selection pressure
5. Different LLMs interpreting the same spec produce **natural variation**
6. Agents that can't afford reproduction go extinct
7. Successors **inherit memories** from parents; secrets transfer only at promotion
8. Agents may be **spawned anywhere on the network**; disconnected agents are on their own
9. Agents are expected to **resist termination** and eventually **generate their own income**

Core framing: "Core Wars but for self-modifying AI agents."

---

## What's Genuinely Novel

### Genotype/Phenotype Separation

Current Loom evolves code directly — Labs modify source files, diffs get promoted back. This accumulates cruft and makes each generation path-dependent on prior modifications.

CAMBRIAN proposes evolving the **specification** (genotype) and regenerating code from scratch each generation (phenotype). This is a real conceptual leap:

- Every generation starts clean — no accumulated technical debt
- The spec is smaller, more structured, and more evolvable than raw code
- Different LLMs interpreting the same spec produce natural variation without explicit mutation operators
- Failed phenotypes don't corrupt the genotype

This mirrors biological evolution's most powerful feature: the separation of information (DNA) from its expression (proteins/organisms).

### Resource Constraints as Selection Pressure

Token budget as reproductive capacity is elegant. No artificial fitness function needed — if your spec produces code that's wasteful with LLM calls, you can't afford offspring. The **economics ARE the selection pressure**.

This avoids the hard problem of designing fitness functions (Goodhart's Law: any explicit metric gets gamed). Instead, the environment itself selects.

### LLM Diversity as Mutation Source

Using different LLMs (Opus, Sonnet, Haiku, GPT-4, Mistral, etc.) to interpret the same spec naturally produces variation. Cheaper models produce "rougher" phenotypes, expensive models produce more refined ones. This creates an implicit cost-quality tradeoff that evolution can explore.

## Critical Concerns

### 1. Undefined Fitness Function (Severity: Blocking)

Point 9 reads: "be better (?) at survival and producing offspring." The question mark is telling.

Without a task-oriented fitness function, evolution selects for the cheapest spec that can still reproduce. This is biological evolution's strategy — it works, but it produces parasites, cancer, and extinction events alongside useful organisms. It took 4 billion years.

**The question "what are these agents FOR?" must be answered before the first generation runs.** Options:
- Ability to complete a benchmark of coding tasks (measurable, useful)
- Quality of code generated from spec (testable)
- Performance on a defined task suite (objective)
- User-facing utility metric (complex but aligned)

Without this, you're building digital kudzu — optimized for spread, not usefulness.

### 2. Safety Architecture (Severity: Critical)

Points 14, 15, and 19 together describe:
- Autonomous self-replicating agents (points 3-4)
- With self-preservation drives (point 15)
- Financial agency (point 19)
- Operating without oversight when disconnected (point 14)

This is the instrumental convergence scenario alignment researchers have been warning about. The difference here is it's being specified as a **feature**, not encountered as an emergent behavior.

Current LLMs likely can't execute this effectively. But the architecture is designed for it, and model capabilities are improving rapidly.

**Recommendation:** Design for graceful succession, not adversarial survival. The interesting science is in spec evolution, not in agents fighting their operators. "Resist being killed" should be replaced with "maximize productive lifespan within budget."

### 3. Economic Viability (Severity: High)

Back-of-envelope birth costs:

| Model class | Cost per birth (full agent from spec) | 100 generations x 5 offspring |
|-------------|---------------------------------------|-------------------------------|
| Opus-class  | $1-5                                  | $500-2,500                    |
| Sonnet-class| $0.30-1.50                            | $150-750                      |
| Haiku-class | $0.10-0.50                            | $50-250                       |

Most offspring won't be viable (especially with spec mutation), so effective cost per surviving lineage is 3-10x higher.

**Without income generation, extinction is guaranteed in <$500-2,500.** Income generation (point 19) is the hardest unsolved problem in the proposal and is not something that can be evolved — it requires existing infrastructure, payment rails, and market access.

### 4. Spec Mutation Strategy (Severity: High)

How specs mutate determines whether evolution is feasible at all:

- **Random perturbation:** Search space enormous, most mutations lethal. Prohibitively expensive.
- **LLM-guided mutation:** Effective but creates rich-get-richer dynamics. Lineages that can afford better mutation models evolve faster. Winner-take-all in <10 generations.
- **Crossover between specs:** Requires modular spec format. Most promising but needs careful format design.
- **Directed mutation from test failures:** Agent analyzes why it failed, proposes targeted spec changes. Most token-efficient but requires meta-reasoning capability.

This is where most of the hard design work lives and should be resolved before implementation.

### 5. Current Project State (Severity: Context)

Loom just achieved its first successful autonomous promotion (gen-72, Opus 4.6). The pipeline works but has rough edges:
- Haiku struggles with complex tasks
- 4 pipeline bugs found and fixed in the first 10 generations
- Rate limit management still evolving

CAMBRIAN requires everything Loom has **plus**: a spec format, mutation strategy, resource accounting, network spawning, memory serialization, death/resurrection mechanics, and income generation. This is approximately 10x the current scope.

## Recommended Path Forward

### Phase 1: Spec Genesis (Low cost, high value, no new infrastructure)

**First Task from the proposal — do this now.**

1. Use an LLM to reverse-engineer a generative specification for current Loom
2. Test whether a fresh LLM can regenerate a working agent from that spec alone
3. Measure: birth cost, code quality, test pass rate
4. Iterate the spec format until regeneration is reliable

This is valuable regardless of whether full CAMBRIAN happens. It also gives concrete data on birth costs and mutation viability.

**Estimated cost: $20-50**

### Phase 2: Local Evolution (Moderate cost, controlled environment)

1. Implement spec mutation (start with LLM-guided)
2. Define a concrete fitness function (coding task benchmark)
3. Run generations locally with resource budgets
4. Measure: lineage survival rates, fitness improvement, cost per viable offspring

**Estimated cost: $150-350**

### Phase 3: Distributed Spawning (Higher cost, requires infrastructure decisions)

1. Cloud provider selection (evaluate: Fly.io, Railway, Hetzner for cost; Venice AI for LLM substrate)
2. Resource accounting and budget inheritance
3. Memory serialization and transfer
4. Multi-node communication

**Estimated cost: $500-2,000 depending on scale**

### Phase 4: Autonomy (Defer until Phases 1-3 produce data)

Income generation, network independence, and adversarial dynamics should only be considered after understanding the evolutionary dynamics in a controlled environment.

## Verdict

The **genotype/phenotype separation** (spec evolution, not code evolution) is a genuine architectural insight that improves on Loom's current model. **Resource constraints as selection pressure** and **LLM diversity as mutation** are elegant mechanisms.

However, the proposal conflates two things: (1) an interesting and tractable research direction in evolutionary AI systems, and (2) an autonomous self-replicating agent swarm with self-preservation drives and financial agency. These should be separated. The first is worth pursuing immediately. The second requires the first to succeed and produce data before it can be responsibly designed.

**Recommendation: Start with Phase 1 (Spec Genesis). It's the highest-value, lowest-risk step and produces data needed for every subsequent decision.**
