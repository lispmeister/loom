## Context

The MVP pipeline is proven stable (5/5 generations, gen-13 through gen-17, zero failures). Gen-21 demonstrated that a Lab can modify agent code using `edit_file`. However, gen-21 also revealed that Haiku (the Lab model) can miss edit sites on complex multi-edit tasks — it changed 3 of 5 required locations. The existing `verify_generation` (tests only) would not catch this kind of incomplete work if the code still compiles.

The next milestone is making the system genuinely recursive: Prime autonomously proposes improvements, Labs execute them, and Prime verifies and promotes — without human intervention.

---

## Design: Three Sub-Phases

### Sub-Phase A: LLM-Powered Verification

**Problem:** `verify_generation` runs `npm test` — nothing more. A Lab can introduce incomplete changes, subtle regressions, or cosmetic-only modifications that pass all tests but don't fulfill the program.md contract.

**Solution:** After tests pass, Prime sends the diff and the original program.md to its own model (Sonnet or Opus) for code review.

**Why Prime, not Supervisor:**
- Supervisor is infrastructure — it manages containers and git. It should not make judgment calls about code quality.
- Prime already runs `verify_generation` and has access to the stronger model.
- This preserves the architecture: Supervisor = infrastructure, Prime = judgment, Lab = execution.

**Verification flow (updated):**

```
verify_generation(gen-N):
  1. git diff --stat master...lab/gen-N          (what changed)
  2. git diff master...lab/gen-N                 (full diff)
  3. git checkout lab/gen-N
  4. npm test && node out/test.js                (safety gate)
  5. Return to master
  6. NEW: Load programs/gen-N.md                 (original task)
  7. NEW: Send to Claude: diff + program.md +
         "Is this complete and correct?"
  8. NEW: Return structured verdict:
         {:approved bool
          :issues ["..."]
          :confidence :high/:medium/:low
          :summary "..."}
```

**Promotion decision:** Tests must pass AND LLM must approve. If LLM disapproves (even with passing tests), Prime rolls back. This catches:
- Incomplete edits (gen-21's missed edit sites)
- Cosmetic-only changes that don't address the task
- Subtle regressions that tests don't cover

**Verdict is stored in gen-N-report.json** alongside token-usage, test-results, and diff-stats. This gives the reflect step data about *why* generations were promoted or rolled back.

### Sub-Phase B: The Reflect Step

**Problem:** Prime has no agency. It waits for user input, mechanically executes what it's told. For autonomous self-improvement, Prime must decide what to work on next.

**Solution:** After each promote or rollback, Prime enters a `reflect` phase that generates the next program.md.

**Reflect inputs:**
1. Last generation report (outcome, LLM verdict, test results, fitness score, token usage)
2. `priorities.md` (user-authored improvement priorities)
3. Generation history (what's been tried, what worked, what failed)
4. Current fitness score and trend

**Reflect output:** A proposed program.md targeting the highest-priority unsolved improvement.

**Design decisions:**

- **Reflect is a Prime tool**, not a standalone component. Prime calls `reflect_and_propose` which returns a program.md. Prime can then review it before calling `spawn_lab`. This preserves the human-in-the-loop option — a user can intercept and modify the proposed program.md.

- **The reflect prompt is the critical design artifact.** It must encode:
  - What was tried and what happened (from generation history)
  - What to prioritize (from priorities.md)
  - What to avoid (don't repeat failed approaches)
  - Scope constraints (single smallest change, not a rewrite)

- **Reflect does NOT read source code.** It works from reports and priorities, not from analyzing the full codebase. This keeps reflect fast and focused. The Lab reads source code when executing the task.

**Anti-pattern to avoid:** Reflect proposing vague or cosmetic improvements ("add more comments", "improve code style"). The prompt must enforce that improvements are measurable and target specific priorities.

### Sub-Phase C: Autonomous Loop Driver

**Problem:** Sub-phases A and B are manual — someone must call reflect, review the program.md, spawn a Lab, verify, promote. For true recursion, this must be automated.

**Solution:** A loop driver that runs the full cycle continuously until a stopping condition is hit.

**Loop iteration:**
```
1. reflect_and_propose → program.md
2. spawn_lab(program.md) → wait for Lab
3. verify_generation(gen-N) → tests + LLM review
4. If approved: promote_generation(gen-N), log fitness
   If rejected: rollback_generation(gen-N), log failure
5. Check stopping conditions
6. Goto 1
```

**Stopping conditions:**
- `LOOM_MAX_GENERATIONS` — hard cap on total generations
- `LOOM_TOKEN_BUDGET` — cumulative token limit across all generations
- Fitness plateau — no improvement over N consecutive generations
- SIGINT — graceful shutdown after current generation completes

**Fitness log:** Append-only JSONL file (`tmp/fitness-log.jsonl`) with one line per generation:
```json
{"generation": 22, "fitness-score": 1161, "tests-run": 101, "assertions": 239,
 "token-usage": {"input": 5000, "output": 300}, "outcome": "promoted",
 "llm-approved": true, "timestamp": "2026-03-16T..."}
```

This enables plotting fitness over time and detecting plateaus.

## Architecture After All Three Sub-Phases

```
User
  │
  ├── Authors priorities.md (what to improve)
  ├── Sets env vars (LOOM_MAX_GENERATIONS, LOOM_TOKEN_BUDGET)
  ├── Starts loop driver
  └── Monitors fitness-log.jsonl, interrupts with SIGINT if needed

Prime (Sonnet/Opus)
  │
  ├── reflect: reads reports + priorities → generates program.md
  ├── spawn_lab: sends program.md to Supervisor
  ├── verify_generation: runs tests + LLM code review
  ├── promote/rollback: based on tests + LLM verdict + fitness
  └── loop driver: repeats until stopping condition

Supervisor (host-side infrastructure)
  │
  ├── Container lifecycle (create, start, stop, destroy)
  ├── Git operations (branch, merge, tag, fetch)
  ├── Generation tracking (generations.edn)
  └── Reports (gen-N-report.json with all metrics)

Lab (Haiku, in container)
  │
  ├── Reads program.md
  ├── Executes task autonomously (edit_file, bash, etc.)
  ├── Commits changes to lab/gen-N
  └── Reports status + token usage via GET /status
```

**Key principle:** Each component has a single role. Supervisor doesn't judge, Lab doesn't verify, Prime doesn't manage containers. The LLM review adds a second layer of verification without changing who owns what.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Reflect proposes vague/cosmetic improvements | Wasted generations, no real progress | Strict prompt: "measurable improvement to a specific priority" |
| LLM review is too lenient (approves bad changes) | Regressions promoted | Require `:confidence :high` for auto-promote; `:medium` triggers human review |
| LLM review is too strict (rejects good changes) | Nothing gets promoted, loop stalls | Track approval rate; if < 30%, tune the prompt |
| Fitness plateau detection triggers too early | Loop stops before meaningful progress | Set plateau threshold to N=5 consecutive non-improvements |
| Token budget exhausted on bad reflect proposals | Cost with no value | Limit reflect itself to small context; cap Lab model to Haiku |
| Lab makes changes that compile but are wrong | Promoted despite LLM review | LLM review prompt must explicitly check completeness against program.md |

## Dependency Graph

```
Sub-phase A (LLM verification)
  ├── A.1: Add LLM diff review to verify_generation
  ├── A.2: Define verdict schema
  ├── A.3: Include verdict in reports
  ├── A.4: Retrieve program.md for verification
  └── A.5: Validate with 2-3 test generations
              │
Sub-phase B (reflect step)  ← depends on A
  ├── B.1: Implement reflect function
  ├── B.2: Design reflect prompt
  ├── B.3: Add as Prime tool
  └── B.4: Test manually with 2-3 cycles
              │
Sub-phase C (loop driver)   ← depends on A + B
  ├── C.1: Implement loop driver
  ├── C.2: Stopping conditions
  ├── C.3: Fitness log
  └── C.4: Run 5 autonomous generations
```

Each sub-phase is independently testable. A failed sub-phase doesn't break the existing pipeline — it just means we can't proceed to the next level of automation.

## Success Criteria

- **Sub-phase A complete when:** LLM review catches at least 1 incomplete change that tests would have missed (like gen-21's case).
- **Sub-phase B complete when:** Reflect proposes 3 consecutive program.mds that target real priorities (not cosmetic).
- **Sub-phase C complete when:** 5 autonomous generations run without human intervention, with at least 2 promotions and measurable fitness improvement.
- **Overall milestone complete when:** We can say "I pointed it at itself and walked away. Here's what it improved." with data to back it up.
