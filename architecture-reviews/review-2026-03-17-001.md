## Schema vs Reality Mismatch

The `Generation` Malli schema in `supervisor/generations.cljs:11-20` defines:

```clojure
[:outcome [:enum :promoted :failed :timeout :in-progress]]
```

But the actual data in `generations.edn` uses `:done` as an outcome, which is **not in the schema enum**. The `valid?` function exists but is never called — no validation runs when generations are appended or updated. The schema is decorative.

This is the same class of issue flagged in review-2026-03-15-001 (finding #2: "Three Malli schemas are defined but never used"). The pattern persists: schemas exist as documentation rather than enforcement.

**Consequence for autonomous loop:** The `run-cycle` function in `autonomous.cljs` decides promote vs rollback based on verify results. But the Supervisor records the outcome as `:done` when the Lab finishes, before verification happens. If the autonomous loop crashes between Lab completion and promote/rollback, the record stays as `:done` — an ambiguous state that's neither promoted nor failed.

## Report Data Gap

Generation reports (`tmp/programs/gen-N-report.json`) capture Lab-side metrics but **not verification data**. Example from gen-24:

```json
{
  "generation": 24,
  "outcome": "done",
  "duration-ms": 53914,
  "token-usage": {"input": 66106, "output": 1769}
}
```

No `test-results`, `diff-stats`, or `llm-verdict` fields. This is because `save-generation-report` in `supervisor/lab.cljs` writes the report when the Lab finishes. Verification data lives in the `last-verification` atom in `self_modify.cljs` and is supposed to be included when promote/rollback POSTs to the Supervisor. But since only gen-1 was ever promoted, 41 out of 42 reports are incomplete.

**Impact on autonomous loop:** The reflect step (`reflect.cljs::gather-context`) reads generation reports to make decisions. Without verification data, reflect has no signal about *why* a generation succeeded or failed. It can only see that the Lab ran and consumed tokens.

The fitness log (`tmp/fitness-log.jsonl`) has only 2 entries (gen-40 and gen-41), both with zero scores and zero test counts. The autonomous loop has barely been exercised, and when it was, it produced no useful fitness data.

## Autonomous Loop Readiness Assessment

**Question: Is the system ready for C.4 (5 autonomous generations end-to-end)?**

**Answer: Not yet.** Several issues would cause the loop to fail or produce meaningless results:

### 1. verify-generation operates on the host repo

`self_modify.cljs:332` runs `git checkout lab/gen-N` on the host repo, runs tests, then returns to master. During an autonomous loop, this means:

- The host repo is temporarily on a lab branch during each verification
- If two verifications overlap (shouldn't happen in the loop, but could with concurrent use), the repo state is corrupted
- If the Supervisor is also running from the same repo, a checkout during verification could disrupt it

This is a known issue from review-2026-03-15-001 (finding #4) that hasn't been addressed.

### 2. No priorities.md content drives empty reflects

The reflect step reads `priorities.md` for guidance. The file exists, but if its content doesn't match what the Lab can realistically accomplish in 5 minutes with Haiku, reflects will propose over-ambitious tasks that fail.

### 3. Token budget tracking is incomplete

The autonomous loop reads token usage from generation reports. But as shown above, reports only contain Lab-side tokens. The reflect step's own Claude API call (building program.md) and the verify step's LLM review call are not counted. The `token-budget` stopping condition undercounts actual consumption.

### 4. Fitness plateau detection requires promoted generations

The plateau detector checks if the last N *promoted* generations have the same fitness score. If no generations get promoted (e.g., verification fails on all of them), `recent-scores` stays empty and the plateau check never triggers. The loop would run until the generation cap.

This is by design (the generation cap is the backstop), but it means that 5 failing generations in a row would all run — burning tokens without any useful signal.

### 5. The case expression for outcome mapping

In `autonomous.cljs:228-231`:

```clojure
(let [outcome-key (case (:outcome result)
                    :promoted     :promoted
                    :rolled-back  :rolled-back
                    :failed)]
```

`:failed` serves as the default case, which correctly handles `:spawn-failed`, `:reflect-failed`, etc. But this is non-obvious — a reader would expect explicit handling of the outcomes that `run-cycle` actually returns. Defensive code should at minimum log unexpected outcomes.

### Dead code (still present)

`shared/eval_client.cljs` (41 LOC) and `lab/eval_server.cljs` (110 LOC) remain in the codebase. Flagged in review-2026-03-15-001 (finding #3). The `:lab` build target still exists in `shadow-cljs.edn`. This is 151 LOC (~4% of source) that compiles but never runs.

### verify-generation promise nesting (still present)

`self_modify.cljs:309-398` is a single function with 8+ levels of `.then` nesting — 90 lines of promise chains. Flagged in review-2026-03-15-001 (finding #9) and review-2026-03-16-003 (implementation critique). Breaking this into named step functions would make the verification flow auditable.

### last-verification atom coupling (still present)

The `last-verification` atom (`self_modify.cljs:65`) is the sole communication channel between verify and promote/rollback. Flagged in review-2026-03-16-003 (finding #2). A verify call overwrites the atom; a subsequent promote reads it. If anything intervenes (concurrent use, crash, out-of-order calls), the data is stale or wrong.

For the autonomous loop this is less risky (sequential execution within a single process), but it's still a fragile pattern that prevents any future parallelism.

1. **Test health is excellent.** 126 tests, 316 assertions, 0 failures. Up from 90/239 at the first review. The test-to-source ratio (~0.73 by LOC) is solid.

2. **Error handling is consistent.** All async operations resolve to data (never throw). The promise chains in `run-cycle` handle each failure mode and continue the loop.

3. **Tool filtering for Labs works.** The read-only bug fix (gen-39) proved that Labs with only 4 base tools produce results. This is a genuine architectural improvement.

4. **Observability is comprehensive.** Dashboards, SSE streams, JSON reports, fitness log, and generation history provide multiple views into system state.

5. **Configuration is flexible.** Environment variables control every tunable: timeouts, budgets, models, endpoints, iteration caps. Switching between providers (Anthropic, Minimax) requires only env changes.

6. **The codebase is lean.** ~3,500 LOC for a working self-modifying agent with LLM verification, reflection, and an autonomous loop driver. The minimalist approach (no frameworks, no macOS-specific APIs beyond containers) pays off in auditability.

## Recommendations

### Before C.4 (blockers)

1. **Run a manual verify → promote cycle on a recent generation.** Gen-42 completed successfully (264s, 226K tokens). Verify it, promote or rollback. This exercises the path the autonomous loop depends on and populates a report with full verification data. Until this works reliably by hand, automating it won't help.

2. **Reconcile orphaned generations.** Mark gen-7, gen-33, gen-40, gen-41 as `:failed` (or add a `:stale` outcome). Add a Supervisor boot-time reconciliation that marks any `:in-progress` generation older than `LOOM_LAB_TIMEOUT_MS` as `:timeout`.

3. **Fix the Generation schema to match reality.** Add `:done` to the outcome enum, or change the code to use `:promoted`/`:failed` at the right lifecycle points. Enforce validation on write.

### After C.4 (hardening)

4. **Merge verification data into reports on promote/rollback.** The Supervisor's promote handler should read the verification payload from the POST body and merge it into the existing `gen-N-report.json`. This ensures reports are complete regardless of when they're read.

5. **Track reflect + verify token usage.** Add the autonomous loop's own API calls (reflect prompt, LLM review) to the token budget calculation. Currently only Lab-side tokens are counted.

6. **Break up verify-generation.** Extract named functions: `get-diffs`, `run-and-parse-tests`, `run-llm-review`, `store-verification`. Each returns a promise. The top-level function becomes a sequential pipeline.

7. **Remove dead code.** Delete `eval_client.cljs`, consider whether `eval_server.cljs` belongs in the build. Remove the `:lab` build target from `shadow-cljs.edn` if unused.

## Addendum: Seed Loop Strategic Assessment

*Added 2026-03-17 after the following prompt from the project architect:*

> Our overall goal is to create the SEED agentic loop that can bootstrap itself through recursive self-improvement. Obviously we are using external LLM thinkers (for now) to create generations (and to evaluate potential improved candidates). The SEED AGENTIC LOOP represents the core of an evolving system that serves the user but also itself as it gains more functionality and is able to create and test new features autonomously. Considering that, what is your assessment of the current state and what do you recommend in addition to your previous architecture and code review?

---

### What You Have

A **reliable execution pipeline** — spawn, execute, verify, promote/rollback — that can modify its own source code. This is genuinely impressive infrastructure. Most self-modification attempts never get past "code that can edit code." You have code that can edit code, test the edits in isolation, and atomically adopt them.

### What You Don't Have

**A feedback loop from outcomes to strategy.**

Right now the system is a **linear pipeline with a human-authored compass** (priorities.md). Reflect reads the priority list top-down, proposes a task, and a Lab executes it. If it fails, reflect says "don't repeat that." If it succeeds, reflect moves to the next priority.

This is not recursion. This is iteration under human direction.

For a genuine seed loop, the system needs to improve **its own ability to improve.** That's the meta-recursive property. Currently:

| Self-X Capability | Status | Why It Matters |
|---|---|---|
| Self-observation | Minimal — test counts, token totals | Can't diagnose what's actually happening |
| Self-evaluation | Weak — fitness = tests\*10 + assertions - tokens/1000 | Proxy that doesn't correlate with real improvement |
| Self-modification | Partial — can edit .cljs files | Can't edit strategy, fitness, tools, or its own prompts |
| Self-selection | Decent — tests + LLM review | Good gate, but gate criteria are invisible to the system |

### The Core Problem

**Reflect is blind.** It can't read its own source code. It can't see the fitness function. It can't observe which tools Labs actually use. It can't analyze why failures happen beyond "outcome: failed." It proposes tasks based on user priorities and test count trends — that's all.

This means the system **cannot propose changes to the things that matter most:**
- The reflect prompt itself (how it decides what to work on)
- The fitness function (how it evaluates success)
- The tool definitions (what capabilities Labs have)
- The verification criteria (what "good enough" means)
- The stopping conditions (when to quit)

These are the control surfaces of the loop. If the system can't see or touch them, it can't improve itself recursively. It can only improve the code that runs *inside* the loop.

### The Bootstrap Path

Think of this like a compiler bootstrap. You hand-write the first compiler in assembly. Then the compiler can compile improvements to itself. The key insight: **the first improvement must be manually orchestrated, and it must give the system the ability to see itself.**

**Step 0: Introspection.** Give reflect the ability to read the codebase. Not just priorities.md and generation reports — the actual source files. Let it analyze `fitness.cljs`, `tools.cljs`, `loop.cljs`, `reflect.cljs`. This is the equivalent of giving the compiler the ability to read its own assembly.

**Step 1: Make the control surfaces data-driven.** Move the fitness function from hardcoded ClojureScript to an EDN config file. Same for the reflect prompt template, the Lab system prompt, and the tool registry. Now these are *data the system can propose changes to* rather than code it can't see.

**Step 2: Add a "lessons learned" accumulator.** After each cycle, write a brief structured analysis: why did this succeed/fail? What class of task was this? What tools did the Lab use? Append to a file that reflect reads on the next cycle. This gives reflect memory — not just "gen-42 failed" but "gen-42 failed because the Lab tried to edit a function that had been refactored in gen-40."

**Step 3: Close the meta-loop.** Let reflect propose changes to itself. "I should also consider code complexity when proposing tasks." A Lab edits `reflect.cljs`. Verification confirms the reflect tests still pass. Promote. Now reflect is genuinely different — and the system chose to make it different.

### Self-Improvement Capability Matrix

| Aspect | Can Modify? | Method | Constraints |
|---|---|---|---|
| Agent code | Yes | program.md task for Lab | Must complete in <5 min, tests must pass |
| Tool implementations | Yes | Edit tools.cljs via Lab | No validation Labs can't break each other |
| System prompts | Yes | Edit loop.cljs or reflect.cljs | Prompt engineering done blind (no A/B test) |
| Test suite | Yes | Add/edit test files | Fitness only counts quantity, not quality |
| Fitness function | **No** | Hard-coded in supervisor/fitness.cljs | Reflect doesn't even see this file |
| Tool definitions | **No** | Could theoretically, but risky | Lab doesn't have context for API changes |
| Schemas | **No** | Locked down explicitly | Fixed-point contracts |
| Supervisor | **No** | Out of reach; Labs run in containers | By design |
| Reflect logic | Partially | Could edit reflect.cljs via program.md | But reflect itself can't propose this |
| Priorities file | **No** | Read-only from Prime's perspective | Only user can edit |

**Translation:** Labs can refactor code and add tests. But the *scaffolding* (fitness, tools, decisions) is locked down or invisible.

### What Reflect Actually Sees vs What It's Missing

**Reflect's inputs:**
- `priorities.md` — user-authored improvement goals
- Last N generation reports — outcome, tokens, test counts, LLM verdict, fitness score
- Current generation number and latest outcome

**What reflect cannot see:**
- Its own source code or prompt template
- The fitness function formula
- Tool definitions or registry
- Agent loop internals (iteration counts, context trimming decisions)
- Lab tool-call frequency and error rates
- Verification depth metrics (code coverage, import checks)
- Why specific failures happened (only "outcome: failed")
- Performance trends (is fitness trending up or oscillating?)
- Its own previous reasoning (what it proposed and why)

### Information Asymmetry: Prime vs Labs

| | Prime Sees | Labs See |
|---|---|---|
| Priorities | Full priorities.md | Nothing |
| History | Full generation history + reports | Nothing |
| Fitness | Scores (but not the formula) | Nothing |
| Source code | Could read it (has read_file) but reflect doesn't | Full repo clone |
| Context | Multi-turn conversation | Only program.md |
| Prior learnings | Test results, verdicts | Nothing |

Labs reinvent the wheel on every task. They can't ask "what did gen-15 try?" or "why did that fail?" They only know the current goal.

### Hardcoded Assumptions Baking Limits Into the System

1. **Test count as proxy for quality.** `fitness = tests*10 + assertions - tokens/1000`. More tests = better. But trivial tests inflate the score without improving the system.

2. **Four tools are enough for Labs.** read_file, write_file, edit_file, bash. What if a task needs local testing, parallel work, or iteration?

3. **Fitness plateau = identical scores.** `(apply = scores)` — exact equality over N generations. No tolerance for noise, variance, or non-monotonic progress.

4. **Reflect always follows priorities top-down.** No exploration budget. No curiosity. No "try something wild." Pure exploitation leads to local optima.

5. **5-minute Lab timeout is fixed.** Complex refactors may need longer. Simple tasks waste budget. No adaptive timeout.

### What Data the System Needs to Gather

To answer the questions that matter after 20+ autonomous generations:

1. **Where does time go?** Phase breakdown: reflect (seconds), spawn+lab (seconds), verify (seconds), promote (seconds). Is the bottleneck in the Lab or in verification?

2. **What types of tasks succeed?** Taxonomy: test-addition, refactor, new-feature, bug-fix, config-change. Which have highest promotion rate?

3. **Why do tasks fail?** Taxonomy: lab-timeout, tests-failed, llm-rejected, empty-changes, reflect-error. What's the dominant failure mode?

4. **Is the system getting better?** Fitness trajectory. But also: token cost per successful promotion (efficiency), average cycle time (throughput), promotion rate over time (hit rate).

5. **Is reflect learning?** Track proposal diversity — are we cycling through the same 3 ideas, or genuinely exploring?

6. **What does the Lab actually do?** Tool call counts per generation. If Labs spend 35 of 40 iterations reading files and 5 writing, that's a signal.

Without this data, running C.4 gives you 5 generations of "it ran" with no insight into whether the loop is converging on anything useful.

### The Strategic Argument

The system today is a **code improver under human direction.** It can reliably modify code, test the changes, and adopt them. But the decision about *what to change* comes from a human (priorities.md) fed through a stateless LLM call (reflect).

To become a **seed loop**, it needs three things it currently lacks:

1. **Introspection** — reflect must be able to read its own source code, see the fitness function, understand what tools exist. Without this, it proposes tasks in the dark. *(loom-1kl)*

2. **Memory** — after each cycle, the system must record *why* things worked or failed, not just *that* they did. Reflect reads these lessons and avoids repeating mistakes. *(loom-z2a, loom-6bt)*

3. **Meta-modifiability** — the control surfaces (fitness function, prompts, tool registry) must be external data files, not embedded code. Then Labs can propose and test changes to the improvement process itself. *(loom-07r, loom-g6p)*

The bootstrap sequence is: **give it eyes (introspection) → give it memory (lessons) → give it hands on its own controls (meta-modifiability)**. Each phase is independently valuable and testable. Phase 4 is where the loop becomes genuinely recursive — a Lab can improve the prompt that guides future Labs.

## Addendum: Dual-Mode Operation — User Value vs Self-Improvement

*Added 2026-03-17 after the following prompt from the project architect:*

> How do these new findings gel with creating an agentic programming environment that still can receive build requests from a user while self-improving its own tooling and its capabilities? The user will only keep it running if the tokens and energy the SEED AGENTIC LOOP consumes also produce useful output. Is that realistic or should we focus on pure self-improvement for now?

---

### The Economic Reality

Each cycle costs ~50K-220K tokens (Lab) plus reflect and verify calls (Prime-side). At current prices that's roughly $0.02-0.10 per cycle. Five autonomous generations burning $0.50 on test-count improvements that no user cares about is a bad trade. The system must produce value to justify its existence.

### The Architecture Already Supports Dual-Mode

The execution pipeline doesn't care where program.md comes from. `spawn_lab` accepts a program.md string — it can come from:

- A user saying "add feature X" in `/chat` → Prime writes program.md → spawns Lab
- The reflect step during autonomous improvement
- A CLI invocation (`node out/agent.js spawn task.md`)

The pipeline (spawn → verify → promote/rollback) is source-agnostic. The tangling is in the *autonomous loop*, which hardwires reflect as the only task source. But that's a thin wrapper — `autonomous.cljs` is 262 lines.

### What's Missing for Dual-Mode

The gaps are small and specific:

1. **No task-source tagging.** Generations don't record "this came from user" vs "this came from reflect." Without this, the system can't learn that user tasks have different success patterns than self-improvement tasks. *(loom-1tb)*

2. **No scheduling.** If a user submits a build request while self-improvement is running, there's no preemption. The user waits for the current autonomous cycle to finish (potentially minutes).

3. **Reflect doesn't learn from user tasks.** If the user asks for "add a REST endpoint" and the Lab fails because `edit_file` can't handle multi-site edits, reflect has no signal that `edit_file` needs improvement. The most valuable self-improvement comes from user task friction — and right now that feedback loop doesn't exist. *(loom-3sf)*

4. **Fitness doesn't measure user value.** test-count is an internal quality metric. "Did the user's task succeed?" is the metric that justifies the system's existence. *(loom-jjx)*

### Recommendation: Neither Pure Self-Improvement Nor Pure User-Serving

**The compiler bootstrap analogy extends here.** GCC didn't start by optimizing itself in a vacuum. It started by compiling user programs. The improvements came from making user compilation better. The self-improvement was *driven by* user needs.

The pragmatic path for Loom:

**Phase 1 (now): Get the pipeline instrumented and reliable.** This is the current P0/P1 work — fix orphaned state, enrich reports, add failure categorization. This benefits both modes and takes days, not weeks.

**Phase 2 (next): Add user-requested generations as a first-class concept.** User describes a task to Prime, Prime writes program.md, Lab executes, Prime verifies. Tag the generation as `source: :user`. This produces immediate user value AND generates data. *(loom-7cv)*

**Phase 3 (then): Wire user task outcomes into reflect for friction-driven self-improvement.** "Users asked for 3 features. 2 succeeded, 1 failed because edit_file couldn't handle the change. Propose: improve edit_file." Self-improvement becomes user-friction-driven. Reflect's job shifts from "work through priorities.md top-down" to "analyze what's failing in user tasks and fix the root cause." *(loom-3sf)*

This is a fundamentally different loop than pure self-improvement:

```
User task → Lab executes → outcome recorded
                              ↓
                    Reflect analyzes friction
                              ↓
              Self-improvement task → Lab executes → verify
                              ↓
                    Tool/capability improved
                              ↓
              Next user task works better ← feedback loop
```

The user sees value from day one (their tasks get executed). The self-improvement is targeted at making user tasks succeed more often. Token spend is justified because each improvement directly increases the system's usefulness.

### Revised Execution Plan

```
PHASE 0-1: Instrument & clean (current P0/P1 beads)
     ↓
PHASE 2a: Source tagging (loom-1tb) ──→ User-requested generations (loom-7cv)
     ↓                                       ↓
PHASE 2b: C.4 autonomous run ←── Generates data from BOTH user + reflect tasks
     ↓
PHASE 3: Friction-driven reflect (loom-3sf)
     ↓
PHASE 4: User success rate in fitness (loom-jjx)
```

### The Key Insight

**The user is your fitness function.** If they keep running the system, it's working. If they stop, no amount of test-count improvement matters. Build the system so that every self-improvement cycle makes the next user task more likely to succeed, and the economics take care of themselves.
